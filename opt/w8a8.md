# W8A8 主线

## 定位

W8A8 是优化演进线。

清晰故事是：

```text
Torch baseline -> PTX for small M -> CUTLASS for large M -> benchmark-driven M buckets
```

目标不是维护很多 runtime 特例，而是展示 FP8 activation + FP8 weight 推理如何从 reference 实现逐步走向专用 kernel。

## 保留

### 1. Torch Baseline

Torch 是正确性和性能的起点。

- 用 `torch._scaled_mm` 作为 W8A8 reference 路径。
- 保留它做质量检查和 baseline microbenchmark。
- 不把它作为优化后的生产路径。

### 2. PTX 小 M 路径

PTX 负责 decode 和 small-batch。

- 先瞄准 `M=1` decode。
- 只有 benchmark 证明值得时，才扩到更多 small M bucket。
- 保持 shape-specialized 和显式实现。

PTX 回答的问题是：

> 当我们自己掌控 kernel 时，small-M W8A8 能做到多快？

### 3. CUTLASS 大 M 路径

CUTLASS 负责 prefill 和 large-M。

- 用在较大的 M bucket。
- 把它当成成熟 template-kernel 路线。
- 和 PTX 对比后决定 bucket 边界。

CUTLASS 回答的问题是：

> 从哪个 M 开始，成熟 GEMM template 比自定义 small-M PTX 更合适？

## 下一阶段优化顺序

下一阶段先优化算子本身，再做性能分桶。原因是当前 bucket benchmark 如果包含 host sync、临时 workspace 分配和额外 launch 等结构性开销，得到的 PTX/CUTLASS 边界会被污染。

优先级：

1. **先清理 CUTLASS hot path 结构性开销**
   - 移除 `x_scale.item()` 和 `w_scale.item()` 导致的 CPU/GPU 同步。
   - activation quantization kernel 从 device pointer 读取 `input_scale`。
   - CUTLASS epilogue 从 device pointer 读取输出 scale，避免 host 端计算 `alpha = input_scale * weight_scale`。
   - 第一版允许用极小 device helper kernel 生成 `alpha_device`，但必须记录这个 launch 的成本；如果 launch 成本明显影响 small/medium M，下一版改成自定义 epilogue 直接读取 `input_scale` 和 `weight_scale` 并相乘。
   - 检查 CUTLASS workspace：`workspace_size == 0` 时不得分配临时 workspace；非 0 时记录大小和是否值得引入复用 buffer。
2. **确认 PTX small-M 路径无 host sync**
   - 确认 PTX/JIT hot path 不读取 device scalar 到 host。
   - JIT 编译必须通过 warmup 排除在 steady-state benchmark 之外。
   - PTX 只承担 decode/small-M，不为追求 large-M 覆盖而扩大复杂度。
3. **只在无结构性同步版本上做 bucket benchmark**
   - benchmark 先比较 `torch`、`ptx`、`cutlass`，再比较 `auto`。
   - bucket 边界必须来自去同步后的结果。
4. **最后做 CUTLASS tile 调优并收敛 auto policy**
   - 调优后的结果写回本文件。
   - 默认 runtime policy 仍然只接受简单 M bucket；不把临时实验性 tile 组合变成隐式 fallback。

## CUTLASS 调优协议

CUTLASS 调优目标是追求 prefill / large-M 的最优 W8A8 GEMM 性能，同时保持结果可复现、可解释、可汇报。调优过程不做宽泛 fallback，不把失败结果静默隐藏。

### 固定输入范围

每轮调优至少覆盖这些 M：

```text
M = 16, 32, 64, 128, 256, 512, 1024
```

必要时补充：

```text
M = 2048, 4096
```

权重形状按真实模型线性层选择，至少覆盖：

```text
attention q/k/v/o projection
mlp gate/up projection
mlp down projection
```

记录时必须写出 `(M, K, N)`，不能只写 layer 名。

### 调优变量

CUTLASS 调优只围绕这些变量展开：

- M bucket：`16/32/64/128/256/512+`
- N bucket：按 `next_pow2(N)` 分为 `4096/8192/16384/32768`
- `ThreadblockShape`
- `WarpShape`
- `Stages`
- `MathOperator`：`OpMultiplyAddFastAccum` 与 `OpMultiplyAdd`
- epilogue scale 方案：`host_alpha`、`device_alpha_kernel`、`device_epilogue_scale`

不要把 layer-name 作为默认 dispatch 维度。layer-name 只能用于分析表，除非某个 projection 的收益巨大、稳定，并且能用一条很短规则解释。

### 候选 tile 家族

先从现有实现中的 tile family 出发，不一次性引入大量组合：

| Family | Intended M | Threadblock | Warp | Stages | Math |
| --- | ---: | --- | --- | ---: | --- |
| `m16_narrow` | 16 | `16x64x128` | `16x64x64` | 5 | FastAccum |
| `m32_narrow` | 32 | `32x64x128` | `16x64x64` | 5 | FastAccum |
| `m64_narrow` | 64 | `64x64x128` | `32x64x64` | 5 | regular |
| `m64_wide` | 64 | `64x128x128` | `64x64x64` | 3 | FastAccum |

| `m256_wide` | 256 | `64x128x128` | `64x64x64` | 3 | FastAccum |

| `large_n` | 512+ | `256x128x64` | `64x64x64` | 3 | FastAccum |

每次只改变一类变量。比如先固定 epilogue 方案，比较 tile；再固定 tile，比较 epilogue scale 方案。

### 可靠性要求

每个候选都必须记录：

- build 是否成功。
- `can_implement` 是否通过。
- correctness：相对 BF16 dequant matmul 的 `max_abs`、`mean_abs`、`cosine`。
- latency：warmup、repeat、均值 ms。
- throughput：TFLOP/s。
- 对比：相对 Torch `_scaled_mm`、PTX、上一版 CUTLASS 的 speedup。
- 失败原因：编译失败、运行失败、数值异常、性能回退、资源占用过高。

性能结论至少满足：

- 同一 case 重复两轮，结论方向一致。
- 关键 winner 与 runner-up 差距小于 `3%` 时，不认为 winner 稳定领先；选择更简单或更通用的 tile。
- 任何 `auto` threshold 只能来自去同步后的 CUTLASS 与 PTX 对比。

### 不做的事情

- 不重新引入 Triton fused W8A8 主线。
- 不新增 BF16/W8A16/dequant fallback 作为性能策略。
- 不用大量环境变量隐藏调度逻辑。
- 不把每个 projection 都写成独立默认策略。
- 不把单次 benchmark 的偶然 winner 直接固化。

## Runtime Policy

forward policy 要保持简单：

```text
backend=torch:
    使用 Torch scaled_mm baseline

backend=ptx:
    使用 PTX 路径

backend=cutlass:
    使用 CUTLASS 路径

backend=auto:
    如果 M <= threshold:
        使用 PTX
    否则:
        使用 CUTLASS
```

threshold 必须来自 benchmark。先用粗 bucket：

```text
M = 1, 2, 4, 8, 16, 32, 64, 128, 256, 512, 1024
```

然后选择能解释数据的最小规则，例如：

```text
M <= 16: PTX
M > 16: CUTLASS
```

除非 benchmark 证明有巨大且稳定的收益，并且规则仍然能简单解释，否则不要引入 layer-name 特例。

## 不进主线

下面这些不再作为 W8A8 的 forward 主线：

- Triton fused W8A8 作为 active route。
- shape-aware fallback 到 BF16 或 W8A16。
- dequant-matmul fallback。
- 很多会悄悄改变 runtime policy 的环境变量组合。
- projection-specific 特例作为默认故事。
- hybrid double-buffer 模式，除非 benchmark 证明额外显存成本值得。

历史数据可以留在提交历史里，但 active docs 和 README 只描述 Torch、PTX、CUTLASS 和 benchmark-selected buckets。

### W8A8 CUTLASS custom epilogue scale

- 目标：在已去除 host sync 的基础上，继续消除单元素 `compute_alpha_kernel` launch，把 `x_scale * w_scale` 合并进 CUTLASS epilogue output op。
- 代码改动：
  - `nanovllm/quantization/csrc/w8a8_cutlass_kernel.cu`
  - 新增 `W8A8DeviceScaleCombination` CUTLASS epilogue output op。
  - output op `Params` 直接持有 `x_scale_ptr` 和 `w_scale_ptr`。
  - CUTLASS kernel 内构造 epilogue output op 时 device-load 两个 scale，并计算 `alpha = max(x_scale, 1e-12) * w_scale`。
  - 删除 `compute_alpha_kernel` 和 `alpha` 临时 tensor。
